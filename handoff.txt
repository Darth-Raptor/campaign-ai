============================================================
1. PROJECT PURPOSE
============================================================

Campaign AI is an Arma 3 operational campaign simulation mod.

The goal is to create an ALiVE-inspired but separately implemented campaign AI system that supports:

- Mission-builder-defined objectives
- Virtual AI force structures
- AI commanders with fog-of-war awareness
- Python-owned campaign state through Pythia
- SQF-side validation and Arma execution
- Weighted virtual combat resolution
- Objective ownership/control simulation
- Commander decision scoring
- Route/path/network analysis
- Local JSON persistence
- Future SQLite support
- Future VPS REST API backend
- Future War Room-style dashboard support
- Future map indexing/objective clustering
- Debug visualization for development and Zeus/admin oversight

Campaign AI should operate at the campaign and operational level.

It should not micromanage squad-level behavior such as stance, formation, suppressive fire, building clearing, or individual soldier tactics.

Arma AI handles local tactical behavior when groups are spawned near players.

Campaign AI handles the larger war.

============================================================
2. CORE DESIGN PRINCIPLE
============================================================

Python is the campaign brain.

SQF is the Arma execution gate.

Python may own:
- campaign state
- commander state
- objective state
- virtual group state
- commander knowledge
- combat calculations
- route scoring
- persistence
- future backend communication

SQF must own:
- Eden module reading

- spawning/despawning live Arma groups
- debug markers

- live mission safety

Python recommends.
SQF validates.
SQF applies.

============================================================
3. NON-NEGOTIABLE ARCHITECTURE RULES
============================================================

1. Campaign AI is server-authoritative.
   - Commander cycles run only on the server.
   - Virtual group state is authoritative on the server/Python side.
   - Objective ownership is authoritative on the server/Python side.
   - Clients may display debug information later, but must not make command decisions.

2. Pythia is a required dependency.
   - Campaign AI should disable itself if Pythia is unavailable.
   - The mission must continue if Campaign AI disables itself.
   - Failure must be logged clearly in the RPT.

3. Python owns major simulation state.
   - SQF should not maintain a competing full simulation state.
   - SQF may keep lightweight mirrors for debug/application only.
   - Python is the source of truth for Campaign AI campaign data.

4. SQF validates all Python recommendations.
   - Python cannot directly manipulate Arma objects.
   - Python cannot directly execute mission logic.
   - Python cannot directly issue live Arma orders.
   - SQF must reject invalid, stale, unsafe, or unauthorized recommendations.

5. Commanders must use fog of war.
   - Commanders must not make decisions from perfect true enemy state.
   - Commanders use commander knowledge state.
   - Sensor/combat systems may compare true state, but commander strategy must not.

6. Combat resolver and commander logic remain conceptually separate.
   - Commander asks: “What should I do?”
   - Combat resolver asks: “What happened when forces met?”
   - Combat resolver must not make strategic decisions.
   - Commander must not directly fudge combat outcomes.

7. Do not build a full ALiVE clone.
   - The first target is a predictable, debuggable conventional commander.
   - Avoid insurgency/civilian systems in V1.
   - Avoid complex backend/dashboard systems in V1.
   - Avoid full map indexing in V1.

8. Debug output is mandatory.
   - Every commander cycle should be diagnosable.
   - Every Python call should be traceable when debug is enabled.
   - Every virtual combat result should produce useful reasoning output.

============================================================
14. SQF ↔ PYTHON API CONTRACT
============================================================

All SQF calls into Python must go through:

CAI_fnc_pyCall

Do not scatter direct py3_fnc_callExtension calls across the codebase.

CAI_fnc_pyCall responsibilities:

1. Check isServer.
2. Check CAI_systemsEnabled where appropriate.
3. Check Pythia availability.
4. Call py3_fnc_callExtension.
5. Validate response envelope.
6. Log Python logs if debug enabled.
7. Return standardized SQF result.
8. Disable Campaign AI on critical failure.

Raw call pattern inside wrapper:

[
    "CAIPython.api.function_name",
    [_payload]
] call py3_fnc_callExtension;

Example public call:

[
    "CAIPython.api.ping",
    []
] call CAI_fnc_pyCall;

17. INITIAL PYTHIA PROOF-OF-CONCEPT
============================================================

The first working test must be boring and technical.

Goal:
Prove Pythia + CAIPython works on HostHavoc Windows dedicated server.

Required POC sequence:

1. Mission starts.
2. Server calls CAI_fnc_checkPythia.
3. CAI_fnc_checkPythia calls CAIPython.api.ping.
4. Python returns success envelope.
5. SQF logs ping result.
6. SQF sends sample state:
   - one objective
   - one commander
   - one virtual group
7. Python stores state in memory.
8. SQF calls CAIPython.api.save_snapshot.
9. Python writes local JSON snapshot.
10. SQF calls CAIPython.api.load_snapshot.
11. Python loads snapshot.
12. SQF logs loaded state summary.
13. If any step fails, Campaign AI disables itself and mission continues.

Do not build commander AI before this works.

Critical rule:
Commander strategy must read commander knowledge, not perfect enemy state.

Bad:
Commander directly checks all enemy groups.

Good:
Commander checks known contacts, suspected areas, and known/suspected objective ownership.

============================================================
24. ORDER TYPES
============================================================

MVP order types:

DEFEND
- Move to or remain at objective.
- Increase/maintain objective control.
- Hold until reassigned.

PATROL
- Move between friendly objectives.
- Generate presence and detection chances.
- Support area security.

ATTACK
- Move toward known/suspected enemy objective.
- Engage if opposing virtual forces are encountered.
- Objective control changes only through combat/control logic.

REINFORCE
- Move to friendly objective under threat.
- Strengthen local defense.
- Assist friendly groups.

RESERVE
- Stay near HQ or rear objective.
- Wait for future tasking.

Later order types:

RECON
WITHDRAW
SCREEN
AMBUSH
LOGISTICS
RAID

Do not implement later order types until MVP is stable.

SQF flow:

1. Server scheduler triggers CAI_fnc_commanderCycle.
2. SQF exits if not server.
3. SQF exits if CAI_systemsEnabled is false.
4. SQF collects observed events since last cycle.
5. SQF calls CAIPython.api.commander_cycle through CAI_fnc_pyCall.
6. Python updates intel decay.
7. Python scores objectives.
8. Python recommends orders.
9. Python returns orders and debug reasoning.
10. SQF validates each order.
11. SQF applies accepted orders to Arma-side mirrors/debug/spawn systems.
12. SQF logs rejected orders if debug enabled.
13. SQF requests debug snapshot if debug enabled.
14. SQF updates markers.

Python commander logic priority:

1. Is an important friendly objective contested?
2. Are critical objectives defended?
3. Is reserve percentage below required level?
4. Are idle groups available?
5. Is there a known/suspected enemy objective worth attacking?
6. Is local force ratio favorable enough?
7. Is the commander overextended?

Keep this logic simple in V1.

============================================================
28. COMBAT RESOLVER
============================================================

Python owns weighted combat calculations.

Combat occurs only when:
- opposing virtual groups are close enough,
- combat is offscreen,
- players are not within active simulation radius,
- both groups are valid,
- neither group is in manual override,
- no live Arma fight is already handling the event.

Player proximity rule:
If players are close enough for live Arma simulation, do not resolve virtual combat for those groups.

Combat power formula:

Combat Power =
Base Strength
× Unit Type Modifier
× Readiness Modifier
× Morale Modifier
× Posture Modifier
× Terrain Modifier
× Fortification Modifier
× Support Modifier
× Surprise Modifier
× Random Friction Modifier

MVP modifiers:

- base strength
- unit type
- readiness
- morale
- posture
- terrain
- fortification
- support
- surprise
- random friction

Randomness settings:

Low:     x0.90 to x1.10
Normal:  x0.85 to x1.15
High:    x0.75 to x1.25
Chaotic: x0.65 to x1.35

Default:
Normal

Outcome bands by attacker/defender ratio:

3.0+       decisive attacker advantage
2.0-2.99   strong attacker advantage
1.25-1.99  attacker advantage
0.80-1.24  even fight
0.50-0.79  defender advantage
below 0.50 strong defender advantage

Result types:

- ATTACKER_DECISIVE_VICTORY
- ATTACKER_VICTORY
- ATTACKER_COSTLY_VICTORY
- ATTACKER_GAINED_GROUND
- STALEMATE
- DEFENDER_HOLD
- DEFENDER_SUCCESSFUL_DEFENSE
- ATTACKER_REPULSED
- ATTACKER_ROUTED
- DEFENDER_WITHDRAWAL
- BOTH_SIDES_BREAK_CONTACT

Combat result affects:

- group strength
- readiness
- morale
- objective control value
- assigned order
- commander knowledge
- contact reports
- combat history

SQF validates and displays results.

Combat changes objective control:

Objective owner changes only when control threshold is reached.
